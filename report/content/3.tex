\section{Preprocesado}
\subsection{Descarga de datos y modelos}

Para garantizar la reproducibilidad y facilitar el despliegue del proyecto, se ha implementado un sistema automatizado de gestión de recursos siguiendo principios de diseño \texttt{SOLID} y \texttt{Clean Code}. Este sistema se encarga de la descarga y organización tanto del dataset como de los modelos de lenguaje.

\subsubsection{Arquitectura del Sistema de Ingestión}

La arquitectura se ha desacoplado para que cada dominio (datasets y modelos) gestione sus propios recursos:

\begin{itemize}
    \item \textbf{Gestión de Modelos (\texttt{.models/}):} Se ha centralizado la descarga de modelos en un directorio oculto en la raíz del proyecto. La lógica está encapsulada en \texttt{app/models/}, permitiendo descargar archivos GGUF y configuraciones de tokenizadores de forma independiente.
    
\lstinputlisting[language=Python, firstline=3, lastline=19, caption=Lógica de descarga de modelos en app/models/downloader.py]{app/models/downloader.py}

    \item \textbf{Gestión de Datasets (\texttt{app/dataset/}):} Se encarga de la clonación del repositorio de datos mediante Git, manteniendo una interfaz abstracta para asegurar la integridad de los datos originales.

    \item \textbf{Conversores Especializados (\texttt{app/converters/}):} Módulos dedicados a la transformación de formatos, incluyendo la conversión de PDF a Markdown y el formateo de diálogos de SGD al formato estándar de mensajes (\textit{role/content}).
\end{itemize}

\subsection{Estructuración de Diálogos (JSON Chat Format)}

Antes de la tokenización, se ha introducido una fase de \textbf{Estructuración}. Los archivos originales del dataset SGD son complejos y contienen metadatos no necesarios para el entrenamiento directo. Se transforman a un formato de JSON simplificado donde cada diálogo es una lista de mensajes con roles \texttt{user} y \texttt{assistant}. Estos archivos se almacenan en \texttt{.data/structured/}, creando un formato intermedio universal.

\subsection{Tokenización y Chat Templates}

La tokenización convierte el texto en secuencias numéricas. Dado que se utilizan modelos \textit{Instruct}, el preprocesado aplica \textbf{Chat Templates} de Hugging Face. Este paso inyecta tokens de control específicos (como \texttt{<|im\_start|>} para Qwen o \texttt{<start\_of\_turn>} para Gemma) para que el modelo reconozca la estructura de la conversación.

Para maximizar el rendimiento en hardware Apple Silicon, el proceso se ejecuta utilizando el backend \texttt{MPS} (Metal Performance Shaders), moviendo los tensores resultantes directamente a la GPU del dispositivo.

\subsection{Truncado y Padding}

Se aplica una estrategia de truncado para asegurar que ninguna secuencia exceda los 512 tokens. Complementariamente, se utiliza \textit{padding} para uniformizar el tamaño de los lotes, permitiendo la creación de tensores rectangulares eficientes para el entrenamiento.

\subsection{Manejo Secuencias Largas}

Se han seleccionado modelos con soluciones nativas para contextos extensos (Atención Híbrida en Gemma 4 y GDN en Qwen 3.5). Además, se ha implementado una \textbf{ventana de contexto} que limita el procesamiento a los $N$ turnos más recientes del diálogo, preservando la información crítica para la generación de la respuesta.
