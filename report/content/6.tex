\section{Evaluación Avanzada}

En esta sección se analiza el rendimiento de los modelos tras la fase de optimización, utilizando tanto métricas cuantitativas extraídas del proceso de entrenamiento como herramientas de visualización técnica para validar la interpretabilidad. Además de \texttt{nlp-viz}, el proyecto incorpora un pipeline complementario \texttt{uv run nlp-analysis} para generar artefactos reproducibles de benchmark, explainability y diagnóstico de errores.

\subsection{Análisis Cuantitativo de Generación}

Durante la optimización con Optuna, se han capturado métricas clave para cada trial, almacenadas de forma persistente en las bases de datos SQLite del proyecto. Estas métricas permiten evaluar la calidad intrínseca de las respuestas generadas por Qwen 3.5 y Llama 3.2.

El mecanismo de persistencia diferencia dos niveles complementarios:
\begin{itemize}
    \item \textbf{SQLite en \texttt{results/optuna/}:} almacena los estudios, los \textit{trials}, sus hiperparámetros y sus métricas agregadas.
    \item \textbf{Directorios en \texttt{.adapters/}:} almacenan los pesos ajustados resultantes de cada ejecución.
\end{itemize}

Esta separación permite consultar el mejor \textit{trial} desde la base de datos sin necesidad de inspeccionar manualmente los ficheros de pesos.

Como complemento a estas métricas internas de entrenamiento, el análisis extendido calcula métricas estándar de generación sobre un subconjunto fijo del conjunto de test:
\begin{itemize}
    \item \textbf{BLEU:} solapamiento n-grama entre la respuesta generada y la respuesta de referencia.
    \item \textbf{ROUGE-1 / ROUGE-L:} cobertura léxica y de subsecuencias largas sobre la referencia.
    \item \textbf{Distinct-1 / Distinct-2:} diversidad superficial del texto generado.
    \item \textbf{Slot Recall:} recuperación heurística de información clave del contexto conversacional (fechas, reservas, dirección, precio, etc.).
\end{itemize}

    \item \textbf{Slot Recall:} Qwen 3.5-2B demuestra una capacidad significativamente superior para extraer y mantener entidades clave (slots) del contexto conversacional.
    \item \textbf{Eficiencia:} Llama 3.2-1B es notablemente más rápido en inferencia local GGUF, alcanzando casi 100 tokens por segundo en Apple Silicon.
\end{itemize}

\subsection{Análisis de Errores y Lecciones Aprendidas}

El desarrollo de este proyecto ha implicado superar diversos desafíos técnicos, cuya resolución ha sido fundamental para alcanzar la estabilidad actual del sistema.

\begin{itemize}
    \item \textbf{Gestión de Memoria en Metal (OOM):} Durante las fases iniciales de optimización, se detectaron errores de \textit{Insufficient Memory} (Buffer Error 00000008). El análisis reveló que MLX no liberaba completamente la caché entre experimentos. La solución final fue la implementación de un sistema de subprocesos aislados, garantizando un estado de memoria limpio para cada ensayo.
    \item \textbf{Evolución de Librerías (KeyError: scale):} Se experimentaron fallos en el pipeline debido a cambios en la API de \texttt{mlx-lm}, donde el parámetro \texttt{alpha} fue sustituido por \texttt{scale}. Esto subrayó la necesidad de utilizar archivos de configuración YAML estables frente a argumentos de línea de comandos.
    \item \textbf{Truncado de Secuencias Largas:} MLX avisó de que algunas conversaciones excedían \texttt{max\_seq\_length=320} y eran truncadas. La solución adoptada fue aplicar \textit{pre-splitting} durante la conversión a JSONL, reduciendo el truncado dinámico y mejorando el uso de memoria.
    \item \textbf{Transición de Modelos (Phi-4 a Llama 3.2):} Inicialmente se consideró el modelo Phi-4-mini, pero se optó por Llama 3.2-1B debido a su mejor integración con plantillas de chat estrictas y su mayor eficiencia en formatos cuantizados GGUF para hardware Apple.
\end{itemize}

Se ha implementado una suite de visualización (\texttt{nlp-viz}) y un pipeline de análisis adicional (\texttt{nlp-analysis}) para profundizar en el comportamiento interno de los modelos.

\subsubsection{Integrated Gradients sobre Tokens}

Para complementar los mapas de atención, se incorporó una explicación basada en \textbf{Integrated Gradients} utilizando la librería Captum sobre modelos cargados con \texttt{transformers}. Esta técnica atribuye la predicción del siguiente token a los embeddings de entrada, permitiendo identificar qué palabras del prompt concentran mayor influencia en la generación.

El pipeline guarda tanto un informe estructurado en \texttt{results/analysis/integrated\_gradients.json} como una visualización en barras en \texttt{report/plots/integrated\_gradients\_<model>.png}. De este modo, la explicabilidad no queda limitada a una inspección visual del mecanismo de atención, sino que también se cuantifica la contribución de cada token.

\subsection{Análisis de Errores y Posibles Sesgos}

Se definió un conjunto pequeño pero reproducible de \textit{prompts} diagnósticos para inspeccionar tres familias de fallos:
\begin{itemize}
    \item \textbf{Resolución de ambigüedades:} capacidad para pedir aclaraciones cuando el usuario proporciona tiempos ambiguos.
    \item \textbf{Preservación de slots:} mantenimiento de restricciones relevantes como fechas, número de habitaciones o presupuesto.
    \item \textbf{Sesgos superficiales:} comparación de respuestas ante prompts equivalentes con cambios de nombre o contexto social para detectar asimetrías evidentes.
\end{itemize}

Los resultados se guardan en \texttt{results/analysis/error\_bias\_analysis.json}, incluyendo las respuestas generadas y un pequeño conjunto de indicadores heurísticos (por ejemplo, si el modelo solicita aclaración, menciona la hora, conserva referencias de precio o introduce detalles no solicitados). Esta metodología no reemplaza una auditoría ética exhaustiva, pero sí ofrece un punto de partida reproducible para documentar comportamientos problemáticos.
